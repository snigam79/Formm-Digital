Step-by-step: what changed and why
1
Today Card replaces the greeting
The "Welcome back — Pick a feature below" is gone. The Today Card shows: date, personalised greeting, readiness zone (colour-coded), one-line explanation, and a concrete session target in AU + time + RPE. This answers the primary question every returning user has.
GET /api/readiness/today → {zone, score, recommendation, target_au_min, target_au_max, est_duration_min, est_rpe}
2
Stat tiles show status, not counts
Replaced: Day Streak / Workouts / This Week / Progress — all zeros for new users. Two primary CTAs only
Analyse My Form (teal — primary) and Ask My Coach (neutral — secondary). These are the two actions that deliver the most value. Everything else is one tab away. The 7-card feature grid is removed from the dashboard — it belongs in the nav, not the home screen.
4
Form insight card appears after 2+ analyses
Shows the user's most-analysed movement, its trend percentage, and key findings. This rewards consistent use, communicates longitudinal value, and gives users a reason to check the dashboard daily even if they're not training.

Design decisions for this screen
1
Score with a verdict sentence, not just a number
84/100 alone means nothing. "Strong — a few things to improve" tells the user immediately how to feel about 84. The verdict sentence has a fixed vocabulary: Excellent / Strong / Developing / Needs work. Never just a number.
Score buckets: 90–100 = Excellent. 75–89 = Strong. 60–74 = Developing. <60 = Needs work.
2
Each metric score has a colour-coded bar
Form Accuracy: 88 (green bar). Range of Motion: 71 (amber bar). Users can visually identify their weakest metric instantly without reading numbers. Green = above 80. Amber = 60–79. Red = below 60. The bars communicate more than the numbers.
3
"What to work on" replaces generic fault labels
The current product likely shows fault detection labels. This redesign pairs every fault with a specific, actionable fix in plain language. "Depth — slightly above parallel" + "Try ankle circles before squatting." Not just "depth issue detected." The fix must be doable in the next session.
4
"Ask coach about this →" CTA at the bottom
The result screen hands off directly to the coaching chat, pre-loaded with this specific analysis as context. This is the most natural moment for a coaching interaction. "I just analysed your squat (84/100). The depth issue is likely an ankle mobility limitation — here's how to address it in your next session."
On tap: open Coach with system context = {analysis_id, movement, score, faults[]}. Coach opening message references this data automatically.
5
Show comparison to personal baseline after 3+ analyses
After 3 analyses of the same movement, add a comparison line: "↑ 12% vs your first squat analysis." This is the longitudinal payoff that makes recurring use feel rewarding. It's the most powerful retention mechanism in the form check flow.

Key design decisions
1
Merge Face Scan into this screen — not a separate page
The current Face Scan is a standalone page that users navigate to separately. It should be the top CTA on the Readiness screen: "Tap to scan — updates your readiness score." After scanning, the score updates on the same screen. No navigation required.

HRV, Wellness, and Load Factor each have their own card with a score, a plain-language description of what it means today, and a progress bar. Tapping any card expands to an explanation. This is how users learn sports science — by seeing how each variable affects their recommendation.

As the user moves any slider, the composite score and zone update immediately (debounced 150ms). This is the most important interaction on this screen — it teaches users that their subjective inputs matter and builds the habit of taking the check-in seriously.

Amber state specifics
1
"What to modify today" replaces generic caution text
Don't just say "train carefully." Tell the athlete exactly what to change: reduce top set weight 15–20%, skip AMRAP finishers, add mobility. These are specific, actionable modifications that a recreational athlete can apply immediately without needing to think too hard.
2
Override mechanism is visible but secondary
The override option is at the bottom, in a subdued style. It exists because athletes must be able to train on amber days — the recommendation is guidance, not a gate. But it requires an explicit choice rather than being the default path. The override tap should log the event for the learning loop.

3
Identify which factor caused the amber — highlight it
In this case the HRV and Load Factor cards have an amber border, while Wellness is green. The user can see immediately that it's their training load + a slightly suppressed HRV causing the caution — not their wellness. This teaches them what to address.

Coach implementation details
1
System prompt includes all user context
Before any user message, the AI receives a structured system prompt containing: readiness score + zone, ACWR + zone, last 3 form analyses (movement, score, top faults), Scout Score + level, current training streak, and any active RTP protocol. The coach should never ask for information Formm already has.
System prompt template: "You are the Formm sports science coach. User: {name}. Today: readiness {score}/{zone}, ACWR {val}/{zone}, last form check: {movement} {score}/100, faults: [{list}], Scout Score: {val}/{level}. Respond as a knowledgeable sports science coach. Be direct, specific, and evidence-based."
2
Opening message is generated from data, not static copy
The first message the user sees is not a welcome screen — it's the coach's first contextual observation based on the data. It should reference something specific: their readiness state, their last form check, or their ACWR. If they just completed a form analysis, it opens referencing that result.
3
Dynamic prompt chips based on current state
The 4 suggestion chips are generated server-side from the user's current data state. Logic: if ACWR amber → "How do I manage my load this week?" If last form check > 7 days ago → "When should I do my next analysis?" If readiness red → "What should I do instead of training today?" If Scout Score just changed level → "What does my new level mean?"
GET /api/coach/suggestions → [{id, label, prompt_text}] · 4 items maximum · generated daily
4
Conversation persists across sessions
Conversation history is stored so returning to the coach feels continuous, not a fresh start. The coach can reference past conversations: "Last week you asked about squat depth — how did the ankle mobility drills go?" This is what bespoke means. Store a rolling 30-day conversation window.

14 — Dev Handoff

Developer Handoff Notes

The specific technical decisions that affect what the design looks and feels like in production. These are the things that get lost between design and development.

Critical implementation rules
R1
Readiness score must count up on render
The score number should animate from 0 to its value over 600ms using an ease-out counter. The ring should fill simultaneously over 900ms. Both must wait for data to load before starting — do not animate placeholder/zero states.
const countUp = (target, duration) => { ... easeOut interpolation ... }
Call after API response, not on component mount.
R2
Wellness sliders must update readiness score in real-time
As sliders move, recalculate the readiness composite client-side (using cached HRV and ACWR values) and update the score display. Debounce at 150ms. Zone colour crossfades at 200ms when zone changes. POST to server only on "Confirm check-in" tap.
R3
Scout Score lock screen shows real progress, not mocked data
The progress bar fill % = (form_analyses_count / 5) × 100. The "X of 5 analyses" text is dynamic. If the user completes their 5th analysis while on this screen, the screen should transition to the unlocked state with an animation — do not require a page refresh.
R4
Coach chat context card is server-generated, not client-assembled
Do not build the coach context card in the frontend from individual API calls. GET /api/coach/context should return a single pre-assembled object with all four data points. This prevents the card from partially rendering with stale data if individual calls return at different times.
R5
All animations wrapped in prefers-reduced-motion
Every CSS animation and JS-driven transition must be wrapped: @media (prefers-reduced-motion: no-preference) { ... animation code ... }. Users with this preference set get instant state changes. This is an accessibility requirement — test it explicitly before shipping.
R6
WorkManager jobs fire at correct local time, not UTC
Morning check-in notification fires at 7:00am user local time. RPE notification fires T+30 min after session_end_timestamp in user local time. Store user timezone on registration. On Indian devices: IST = UTC+5:30. Test on devices set to IST specifically — this is where the NCR target market sits.
